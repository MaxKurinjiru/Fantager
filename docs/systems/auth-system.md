# Auth System

Reference: [arch-spec.md](../arch-spec.md#security-requirements)

Purpose: Document authentication, registration, session management, and authorization strategy.

## Overview

The auth system handles player identity and access control. The player selects a Kingdom (game world) as part of the registration form. After email verification, the player is automatically assigned a random available NPC team in their chosen Kingdom and redirected to Team Dashboard.

## Planned Approach

- **Session-based authentication** (Symfony Security component)
- **Registration**: Email, password + Kingdom selection → email verification required → random NPC team assigned on activation
- **Login**: Standard session login with "remember me" option
- **Session scoping**: `kingdom_id` from the authenticated user scopes all game data; no multi-kingdom switching (a player belongs to exactly one Kingdom).
- **Authorization**: Role-based (ROLE_PLAYER, ROLE_ADMIN); resource ownership checks via voters
- **API auth**: Session-based (shared cookies); internal API only, no external consumers
- **Future**: OAuth2/social login; JWT for public API when exposed externally

## CSRF Protection

All state-changing forms (login, register, password reset) and Stimulus AJAX calls must include a valid CSRF token.

### Implementation

- **Package**: `symfony/security-csrf` (part of `symfony/security-bundle`)
- **Configuration**: Enabled globally via `framework.csrf_protection: true` in `config/packages/framework.yaml`

### Twig Forms (server-rendered)

Each form includes a hidden `_csrf_token` field generated by `csrf_token('intent_name')`:

| Form | Token intent | Field name |
|------|-------------|------------|
| Login | `authenticate` | `_csrf_token` (Symfony Security default) |
| Register | `register` | `_token` |
| Password Reset (request) | `password_reset_request` | `_token` |
| Password Reset (confirm) | `password_reset_confirm` | `_token` |

Template usage (login — uses Symfony Security's expected field name):
```twig
<input type="hidden" name="_csrf_token" value="{{ csrf_token('authenticate') }}">
```

Template usage (register, password reset — custom controllers validate manually):
```twig
<input type="hidden" name="_token" value="{{ csrf_token('register') }}">
```

### Stimulus AJAX (internal API)

For AJAX calls from Stimulus controllers:
1. A `<meta name="csrf-token" content="{{ csrf_token('api') }}">` tag is rendered in the base layout
2. Stimulus controllers read the meta tag and include the token as `X-CSRF-Token` header on every non-GET request
3. Backend validates the header via `CsrfTokenManager::isTokenValid()`

### Validation

Controllers validate tokens before processing. Invalid/missing tokens result in HTTP 403.

## Rate Limiting

Rate limiting protects auth endpoints against brute-force attacks, credential stuffing, and abuse.

### Implementation

- **Package**: `symfony/rate-limiter`
- **Configuration**: Defined in `config/packages/framework.yaml` under `framework.rate_limiter`
- **Login throttling**: Built-in Symfony Security login throttling via `login_throttling` in the firewall

### Policies

| Endpoint | Limiter name | Policy | Limit | Window | Key |
|----------|-------------|--------|-------|--------|-----|
| `POST /login` | (built-in) | sliding_window | 5 attempts | 15 minutes | IP + email |
| `POST /register` | `register` | fixed_window | 10 requests | 1 hour | IP |
| `POST /password-reset` | `password_reset` | fixed_window | 5 requests | 1 hour | IP |

### Response on limit exceeded

- HTTP 429 Too Many Requests
- `Retry-After` header with seconds until window resets
- JSON error body for API calls:
  ```json
  {
    "error": {
      "code": "RATE_LIMIT_EXCEEDED",
      "message": "Too many requests. Try again later.",
      "retry_after": 3600
    }
  }
  ```

## Registration Flow

1. User submits Register form (email, password, confirm password, **game nickname**, **Kingdom selection**)
2. Server validates input (see [00-auth-screens.md](../screens/00-auth-screens.md#form-validation-rules)); validates chosen Kingdom exists and has available capacity; validates nickname uniqueness via `display_name_slug`
3. Server creates User entity with `is_verified = false`, `kingdom_id` set to chosen Kingdom, `locale` set to the Kingdom's `language` (default language inherited from Kingdom), `display_name` as entered, `display_name_slug` as webalized form
4. Server sends verification email (via `symfony/mailer`) with a signed, time-limited token
5. User clicks link in email → `GET /verify-email?token=...`
6. Server validates token, sets `is_verified = true`
7. Server **immediately and automatically** assigns a random available NPC team (`is_npc = true`, `user_id IS NULL`) from the player's Kingdom — sets `user_id` on the team, no extra action required from the player. The inherited team includes **10 heroes** (see [team-system.md](../systems/team-system.md#starting-roster)).
8. User is logged in and redirected to Team Dashboard

**Unverified users cannot log in.** Login attempt by unverified user shows: "Please verify your email first."

### Verification token

- Signed URL (SymfonyCasts VerifyEmailBundle pattern or custom)
- Expiration: 24 hours
- One-time use (invalidated after activation)
- Resend available: rate-limited (same `register` limiter)

## Login/Logout

- `POST /login` — Symfony form_login authenticator; session-based
- `POST /logout` — Destroys session, clears remember-me cookie
- "Remember me" cookie: 30 days, httpOnly, secure, sameSite=lax
- Failed login returns generic "Invalid email or password" (no enumeration)
- Successful login redirects to Team Dashboard

---

## Password Reset Flow

1. User submits email on password reset form
2. Server always responds "If the email exists, we sent a link" (no enumeration on this endpoint)
3. If user exists: sends email with signed reset token (1 hour expiry)
4. User clicks link → `GET /password-reset/confirm?token=...` → shows new password form
5. User submits new password → `POST /password-reset/confirm`
6. Server validates token + new password, updates hash, invalidates token
7. User is redirected to login

---

## Session & API Authentication

- **Web + Internal API**: Both use the same session cookie (no separate JWT)
- **Cookie config**: httpOnly, secure=auto, sameSite=lax (defined in `framework.yaml`)
- **Stimulus AJAX**: Requests to `/api/v1/*` are authenticated via the session cookie automatically
- **Kingdom scoping**: `kingdom_id` on the authenticated user scopes all resource ownership checks and data queries to the correct game world.

## API endpoints (planned)

- `POST /register` — Create account (unverified)
- `GET /verify-email?token=...` — Verify email and activate account
- `POST /login` — Authenticate
- `POST /logout` — Destroy session
- `POST /password-reset` — Request password reset email
- `GET /password-reset/confirm?token=...` — Show new password form
- `POST /password-reset/confirm` — Submit new password
- `GET /api/v1/me` — Current authenticated player info

## Dependencies

- Symfony Security Bundle
- Symfony Mailer (email verification + password reset)
- Doctrine (User entity)
- Kingdom System (kingdom chosen during registration; NPC team assigned on email verification)

## Notes

- Keep the initial implementation minimal: email/password, session, basic roles.
